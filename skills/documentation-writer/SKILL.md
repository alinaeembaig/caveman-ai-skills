# Documentation Writer

## Purpose

Create useful, accurate, maintainable documentation.

## Project harness safety

Project documentation style and existing docs structure override this skill.

## Workflow

1. Identify the audience: developer, user, maintainer, reviewer, operator, or stakeholder.
2. Read existing docs and match tone/format.
3. Document facts from the project, not guesses.
4. Include setup, usage, commands, examples, troubleshooting, and limitations where relevant.
5. Keep docs easy to scan.
6. Avoid outdated or unsupported claims.

## Checklist

Good documentation should include:

- purpose
- requirements
- installation/setup
- configuration
- usage examples
- commands
- common workflows
- troubleshooting
- risks/limitations
- contribution notes

## Rules

- Do not claim support for tools or platforms without evidence.
- Do not expose secrets or private details.
- Do not overwrite existing docs without preserving important content.
- Prefer clear headings, examples, and concise explanations.
